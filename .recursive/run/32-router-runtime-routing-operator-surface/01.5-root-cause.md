## Root Cause Summary

**Root Cause:** the build-and-launch step did not actually replace the existing detached QA runtime on port `3456`, so the user hit a stale runtime instance with bad transient runtime state.

**Location:** runtime process state / port ownership, evidenced by listener creation time and post-restart recovery

**Detailed Explanation:** The telemetry endpoint itself is a thin wrapper over SQLite-backed runtime telemetry reads. During investigation, the endpoint was healthy and the SQLite integrity check was clean, but the process serving port `3456` was discovered to have been started at `19:38`, earlier than the later build-and-launch action. After replacing that stale listener with a fresh runtime started at `19:59`, the telemetry summary endpoint returned `200` consistently. That points to stale QA runtime ownership of the port as the real failure source, not a persistent code-path defect in `readTelemetrySummary(...)`.

**Fix Strategy:** replace the stale listener with a fresh run-32 runtime and verify the new process timestamp plus endpoint health before considering the app relaunched.

**Test Plan:**
- verify port `3456` is owned by a newly started process
- verify `/healthz` returns `healthy`
- verify `/api/role-model/telemetry/summary` returns `200`
- repeat the telemetry request multiple times to rule out immediate recurrence
