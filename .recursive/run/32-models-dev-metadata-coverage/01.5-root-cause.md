## Hypothesis Testing

### Hypothesis 1
**Statement:** The post-refresh regressions are caused by identifier drift: the refreshed normalized catalog no longer contains the stable model ids and preset variant ids that the repo’s runtime fixtures and tests still use.

**Rationale:** The failing registry code requires exact `modelId` matches, the refreshed catalog no longer contains `openai/gpt-4.1-mini-fast` or bare `claude-3.7-sonnet`, and direct backend inspection shows Moonshot now exposes generated variant ids instead of `kimi-code`.

**Test:** Compare fixture model ids against refreshed catalog ids and inspect the live backend Moonshot provider under the same unified-config test setup.

**Result:** Confirmed

**Evidence:**
- `/testdata/router-runtime/fixtures/registry-sources.json` still uses `openai/gpt-4.1-mini-fast` and `claude-3.7-sonnet`
- `/role-model-router/packages/catalog/data/normalized-catalog.json` contains `openai/gpt-4.1-mini` and reseller/provider-scoped Claude ids instead
- backend inspection after refresh shows Moonshot variants `moonshot-api-key` and `moonshot-oauth`

## Root Cause Summary

**Root Cause:** The live `models.dev` refresh path imported raw upstream provider/model identifiers directly, but the repo’s runtime fixtures and compatibility tests still depend on a small local alias/supplement layer that was never merged back into the refreshed snapshot.

**Location:** primary cause in `/role-model-router/packages/catalog/src/refresh.ts`; symptom enforcement in `/role-model-router/packages/endpoint-registry/src/index.ts:301-307` and `/role-model-router/apps/runtime-host-bridge/src/index.ts:4211`

**Detailed Explanation:** Before this run, the repo shipped a tiny curated snapshot containing compatibility aliases and a derived fast model that matched the runtime fixtures exactly. The new refresh command correctly widened coverage to live `models.dev`, but it treated the upstream payload as the complete source of truth and discarded those compatibility IDs. Because endpoint-registry validates runtime sources by exact `modelId`, and because the Moonshot device-authorization flow still referenced the old preset variant ids, the refreshed catalog caused deterministic runtime-state validation failures and a broken Kimi OAuth test path.

**Fix Strategy:** Introduce an explicit local supplement/alias layer that is merged with the live-refreshed snapshot before export. That supplement should preserve stable runtime-facing model aliases and derived local variants required by the repo (`openai/gpt-4.1-mini-fast`, bare `claude-3.7-sonnet`, Moonshot compatibility ids, and similar cases) while keeping live `models.dev` metadata as the default source for the broader provider/model set.

**Test Plan:** Add a failing catalog refresh test that proves live refresh preserves local supplemental alias models, then rerun the full runtime-host suite to verify endpoint-registry validation and Moonshot device authorization recover without losing the newly refreshed provider/model coverage.
