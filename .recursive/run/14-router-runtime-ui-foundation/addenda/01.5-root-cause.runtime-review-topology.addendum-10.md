## Error Analysis

**Reported behavior:** the live runtime review shell broke depending on which backend was fronted, with routes such as `/api/metrics`, `/logs`, and `/api/role-model/runtime/summary` alternating between `404` and `200`.

**Observed runtime behavior:**
- bridge-only routing makes `/api/role-model/*` work but leaves raw host routes unavailable
- host-fronted routing restores raw host routes but drops bridge-owned runtime summary/providers/accounts/endpoints reads
- `/v1/*` downstream behavior depends on the host being attached to a direct bridge instance

**Key insight:** the runtime UI is not a single-backend app. It depends on both the repo-owned bridge and the vendored llama-swap host under one operator origin.

## Reproduction Verification

**Steps:**
1. Start the runtime UI against only the bridge target
2. Load runtime pages that need raw host routes (`/api/metrics`, `/logs`, `/api/version`)
3. Switch to a host-fronted setup and load bridge-owned control-plane routes

**Reproducible:** Yes
**Frequency:** 1/1 during investigation
**Deterministic:** Yes

## Data Flow Trace

1. `runtime-ui` issues requests to both `/api/role-model/*` and raw host routes under one origin
2. `runtime-host-bridge` owns runtime summary, provider, account, endpoint, and controller-style control-plane routes
3. vendored llama-swap owns raw host surfaces such as `/api/metrics`, `/api/version`, `/api/captures/:id`, `/logs`, `/upstream/*`, and `/v1/*`
4. `proxymanager_api.go` proxies only a narrow subset of `/api/role-model/*`, so a host-only front door cannot satisfy the full UI contract

**Failure boundary:** review-time request routing, not UI rendering or provider execution

## Root Cause Summary

**Root Cause:** the live runtime shell depends on a split backend topology, but the review stack was launched as though one backend could satisfy both bridge-owned and host-owned routes.

**Location:** review proxy / runtime topology contract

**Detailed Explanation:** `runtime-ui` expects `/api/role-model/*` and `/healthz` to resolve to `runtime-host-bridge`, while raw host routes and `/v1/*` must resolve to the vendored llama-swap host. The vendored host only proxies a subset of bridge paths, so a single-target launch always breaks part of the app. The correct repair is to keep a split proxy contract and attach the host to a direct bridge via `ROLE_MODEL_BRIDGE_BASE_URL`.

**Fix Strategy:** route `/api/role-model/*` and `/healthz` to the direct bridge; route raw host routes and `/v1/*` to llama-swap; launch the host with an explicit bridge base URL so downstream `/v1` surfaces reflect live runtime endpoints.
