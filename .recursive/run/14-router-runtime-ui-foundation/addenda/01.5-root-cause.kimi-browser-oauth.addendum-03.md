## Error Analysis

**Reported behavior:** After starting Kimi Code OAuth the operator only sees a pending device-auth card with a user code and verification URL, and the binding does not complete unless they manually drive the remaining steps.

**Observed runtime behavior:**
- `runtime-ui` starts device auth and renders the returned `verificationUriComplete`
- the UI exposes a manual `Check connection` action instead of completing the flow automatically
- there is no browser-launch step in the runtime UI
- there is no provider callback URL captured by the host bridge because the upstream Kimi flow is device-code polling, not authorization-code redirect

**Key insight:** The issue is not token persistence or account validation. The issue is that our UI implemented a manual device-code flow, while the upstream Kimi CLI UX auto-opens the verification URL and continues polling until the token is issued.

## Root Cause Summary

**Root Cause:** Kimi Code onboarding was implemented as a manual device-code flow in the runtime UI, while the upstream Kimi flow auto-opens the verification URL and finishes by polling rather than via a redirect callback.
**Location:** `role-model-router/apps/runtime-ui/app/routes/accounts.tsx`
**Detailed Explanation:** The host bridge already returns `verificationUriComplete` and supports token polling. The remaining gap is in the UI orchestration: after device auth starts, the route only renders the URL and waits for a manual refresh action. Because Kimi's upstream device flow does not expose an authorization-code callback URL, expecting a callback completion path is a protocol mismatch. The correct repair is to open the verification URL automatically and keep polling the existing bridge until the account becomes connected, expired, or failed.
**Fix Strategy:** Add runtime-ui behavior that opens the verification URL in a new tab/window on start, starts automatic polling for pending sessions, retains a manual retry button as fallback, and updates the UI copy to reflect that the provider flow is device-code polling rather than callback redirect.
**Test Plan:** Add failing runtime-ui tests for the device-auth helper logic, implement the helper and route integration, then rerun focused runtime-ui build/test plus relevant runtime validators.
