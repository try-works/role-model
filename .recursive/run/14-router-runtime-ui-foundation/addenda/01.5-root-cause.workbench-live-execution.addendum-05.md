## Error Analysis

**Reported behavior:** The workbench always shows a generic result such as `Kimi Code summary`, regardless of the prompt content.

**Observed evidence:**
- prompt `Reply with ONLY the word alpha.` -> `Kimi Code summary`
- prompt `Reply with ONLY the word beta.` -> `Kimi Code summary`

**Key insight:** The workbench is not misrendering a varying provider response. The runtime is currently returning a deterministic fixture body for Kimi execution, so the prompt never reaches a live provider.

## Data Flow Trace

**Symptom location:** workbench result pane
**Bad behavior:** identical generic text for different prompts

**Trace:**
1. Workbench posts to `/v1/chat/completions`
2. Host bridge routes correctly to the activated Kimi endpoint

**Root cause:** execution never performs a live provider HTTP request for Kimi, and the adapter logic was not yet honoring the Kimi model's `openai.chat.completions` request shape.

**Likely cause:** the runtime shell was built for deterministic fixture execution first and had not yet crossed the last seam into live provider execution for OAuth-backed Kimi accounts.

## Root Cause Summary

**Root Cause:** OAuth-backed Kimi workbench requests were still executing against fixture captures, and the OpenAI-compatible adapter ignored Kimi's `openai.chat.completions` request shape.
**Location:** `role-model-router/packages/adapter-execution/src/index.ts`, `role-model-router/apps/runtime-host-bridge/src/index.ts`, `role-model-router/packages/provider-openai/src/index.ts`
**Detailed Explanation:** After the addendum-04 repair, Kimi routing and adapter-family registration were correct, but the runtime execution path still resolved provider responses from static test fixtures. That is why different prompts produced the same assistant text. In addition, the reusable OpenAI adapter still always generated `/responses` requests, while the normalized Kimi model explicitly declares `openai.chat.completions`. A full repair therefore needs both a live provider execution seam and request-shape-aware OpenAI-compatible handling.
**Fix Strategy:** Add an async execution path that can perform live provider requests, use it in the host bridge for OAuth-backed Kimi endpoints with resolved stored access tokens, update the OpenAI adapter to honor `requestShapeHints.providerShape`, and keep deterministic fixture fallback for the older non-live validation surfaces.
**Test Plan:** Add focused regression tests for live routed execution, Kimi chat-completions request shaping, and runtime-host live Kimi execution; rerun focused package tests/builds and the runtime validators.
