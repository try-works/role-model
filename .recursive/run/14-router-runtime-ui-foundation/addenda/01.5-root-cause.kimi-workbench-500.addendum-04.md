## Error Analysis

**Reported behavior:** Workbench returns `Request to /v1/chat/completions failed with 500` when using the Kimi Code endpoint with `moonshotai/kimi-k2.5`.

**Live response body:**
```json
{
  "error": "No provider adapter is registered for ai-sdk-openai-compatible."
}
```

**Failing test evidence:**
- `provider-openai-kimi-compatible.addendum-04.log`
- `runtime-host-kimi-workbench.addendum-04.log`

**Key insight:** The failure occurs after routing selects the Kimi endpoint. The request first crashes during adapter resolution because the chosen provider family is `ai-sdk-openai-compatible` but the runtime only registers `ai-sdk-openai` and `ai-sdk-anthropic`. After fixing that boundary, the same execution path also requires a configured response capture for the Kimi endpoint because the current run-14 host bridge is still fixture-backed for provider execution.

## Data Flow Trace

**Error location:** `packages/adapter-execution/src/index.ts:303`
**Bad value:** `target.adapterFamily = "ai-sdk-openai-compatible"`

**Trace:**
1. `executeChatCompletions()` in `apps/runtime-host-bridge/src/index.ts` routes the Kimi request
2. `executeRoutedRequest()` resolves the chosen endpoint/provider from catalog + account state
3. `resolveExecutionTarget()` returns provider adapter family `ai-sdk-openai-compatible`
4. `resolveAdapter()` searches the registered adapter list and cannot find a matching family

**Root cause:** The host bridge only registers the OpenAI adapter under `ai-sdk-openai`, even though the normalized Moonshot/Kimi provider metadata intentionally uses `ai-sdk-openai-compatible`; and because execution is fixture-backed in this runtime shell, the Moonshot endpoint ids also lacked a configured response capture once adapter resolution succeeded.

## Root Cause Summary

**Root Cause:** The runtime host bridge does not register an adapter for `ai-sdk-openai-compatible`, and the current fixture-backed execution map also omits Kimi endpoint ids, so routed Kimi requests fail with a 500 instead of completing.
**Location:** `role-model-router/apps/runtime-host-bridge/src/index.ts`, `role-model-router/packages/provider-openai/src/index.ts`, `role-model-router/packages/adapter-execution/src/cli.ts`, and `testdata/router-runtime/adapter-captures.json`
**Detailed Explanation:** The Moonshot/Kimi provider is cataloged as `ai-sdk-openai-compatible`, which is correct and distinct from first-party OpenAI. When workbench sends a Kimi request, the routing layer correctly chooses the Kimi endpoint and propagates that adapter family into execution. The first failure happens because the adapter registry only contains `ai-sdk-openai` and `ai-sdk-anthropic`. Once that boundary is repaired, execution continues into the fixture-backed response layer, which still lacks a capture entry for the Moonshot endpoint ids. Since the OpenAI request/response adapter logic is reusable for OpenAI-compatible providers and the host shell already relies on response fixtures, the correct repair is to register the compatible adapter alias and configure Moonshot endpoint captures rather than mutating catalog metadata or special-casing Kimi in the workbench.
**Fix Strategy:** Parameterize or alias the existing OpenAI adapter so it can register under `ai-sdk-openai-compatible`, wire that alias into the runtime host bridge and shared execution CLI, add Moonshot/Kimi response capture entries, and prove Kimi chat execution succeeds with focused regression coverage.
**Test Plan:** Run focused `provider-openai` and `runtime-host-bridge` tests first in RED, implement the alias/registration change, then rerun those tests plus focused host validation commands.
