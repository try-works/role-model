## Data Flow Trace

### Issue A: root command path

**Failure location:** `cmd.exe /c corepack pnpm run schemas:validate` and `cmd.exe /c corepack pnpm run smoke`

**Bad value / bad assumption:** the child script body assumes `pnpm` will be on PATH after `corepack pnpm run <script>` dispatches the root script.

**Call path:**
1. `protocol-fixture-conformance.test.ts` or `gateway-smoke-observability.test.ts` shells out with `cmd.exe /c corepack pnpm run ...`
2. root `package.json` resolves `schemas:validate` or `smoke`
3. the script body launches `pnpm --filter ... exec ...`
4. the child shell cannot resolve bare `pnpm`
5. the conformance test fails before the underlying package command runs

**Root cause:** the root script bodies depend on PATH-provided `pnpm` instead of using a PATH-independent nested invocation.

### Issue B: canonical schema identity

**Failure location:** source canonical schemas under `protocol/schemas/*.schema.json`

**Bad value / bad assumption:** missing `$id` is treated as acceptable because code injects `$id: fileName` after parsing.

**Call path:**
1. schema source files are read from disk without `$id`
2. `packages/schema-tools/src/validate-schemas.ts` loads each schema
3. `loadSchemas()` rewrites the in-memory object to `{ $id: fileName, ...schema }`
4. Ajv compilation and type generation operate on the rewritten object, not the canonical source
5. `packages/conformance/src/schema-test-helpers.ts` repeats the same masking pattern for tests

**Root cause:** the source invariant is violated and both runtime consumers hide that violation instead of failing on it.

## Pattern Analysis

**Working pattern 1:** direct package execution succeeds when the entire invocation starts with `corepack pnpm --filter ... exec ...`, which keeps `corepack` in charge of locating pnpm.

**Working pattern 2:** schema compilation already works once a schema object has a stable `$id`; the missing piece is that the canonical source must provide it instead of post-parse code.

**Broken-vs-working difference:**
- working command path: `corepack pnpm` is present at the exact execution layer that resolves pnpm
- broken command path: a nested bare `pnpm` is invoked after the wrapper hands off to the script body
- working schema identity path: an Ajv-added schema has a stable id
- broken schema identity path: the stable id exists only after code mutation, not in committed source

## Hypothesis Testing

### Confirmed hypothesis A

If the root scripts stop shelling out to bare `pnpm` and instead use a PATH-independent nested invocation, the existing conformance tests should stop failing on command dispatch and return to validating real behavior.

**Supporting evidence:**
- the direct `corepack pnpm --filter ... exec ...` path already succeeds
- the failing logs show the error occurs before the underlying command work starts

### Confirmed hypothesis B

If the canonical schema files declare `$id` in source and the loader/helper stop injecting it, source-level conformance can detect missing identity rather than masking it.

**Supporting evidence:**
- all 19 source files currently omit `$id`
- both loader paths inject `$id` after parsing

## Root Cause Summary

- Root cause for `R1` and `R2`: canonical schemas are incomplete in source, and both schema-loading code paths mask that incompleteness by synthesizing `$id` at runtime.
- Root cause for `R3`, `R4`, and `R5`: root `package.json` scripts use nested bare `pnpm` invocations, which break the tested `corepack pnpm run ...` command path in this Windows environment because the child shell cannot resolve `pnpm`.

## Fix Strategy for Phase 2 Planning

1. Add strict red coverage that asserts canonical schema sources expose stable `$id` values without loader mutation.
2. Add `$id` to every canonical schema file and remove the `$id` injection from both the production loader and the conformance helper, replacing it with explicit source validation.
3. Make the root script bodies PATH-independent by removing nested bare `pnpm` assumptions.
4. Re-run the currently failing conformance slice and the directly related schema-tools tests as green evidence before broader validation.
