## Error Analysis

**Browser symptom:** `Request to /api/role-model/controller failed with 500`

**Direct API error body:**

```json
{
  "error": "No runtime endpoints are available for the runtime host bridge."
}
```

## Reproduction Verification

1. Start the bridge with an empty `decision_only` runtime config.
2. Start the runtime UI against that bridge.
3. Open `/app/control/models`.
4. Observe the shell load while the page-level controller fetch fails with `500`.

Result: reproducible `1/1`, deterministic, and rooted in the controller read path rather than route registration.

## Recent Changes Analysis

- The runtime-config remediation addenda intentionally introduced a valid zero-endpoint pre-activation state.
- The older controller contract still assumed at least one endpoint would always exist.

## Data Flow Trace

1. `ControlModelsRoute` calls `fetchControllerAssignment()`.
2. `GET /api/role-model/controller` reaches `readControllerAssignment()`.
3. The bridge falls through to `getDefaultControllerAssignment()`.
4. `currentRegistry.endpoints[0]` is missing in the valid zero-endpoint state.
5. The bridge throws instead of returning a nullable controller assignment.

## Pattern Analysis

| Aspect | Working case | Broken case |
| --- | --- | --- |
| Runtime config | endpoint available | zero endpoints, valid `decision_only` |
| Controller API | assignment returned | `500` thrown |
| Models page | renders inventory | fails before empty-state rendering |

## Hypothesis Testing

- Hypothesis 1: route registration is broken -> rejected; the shell and route both resolve.
- Hypothesis 2: zero-endpoint controller reads throw in the bridge -> confirmed by direct API reproduction and source inspection.

## Root Cause Summary

- Root Cause: controller state was modeled as mandatory even when the bridge is intentionally configured with zero endpoints.
- Location: `role-model-router/apps/runtime-host-bridge/src/index.ts` and `role-model-router/apps/runtime-ui/app/routes/control-models.tsx`
- Fix Strategy: add failing backend/UI coverage, return a non-throwing empty controller result, and render honest empty/unassigned route states.
- Test Plan: cover the zero-endpoint controller route, nullable controller reads, and refreshed browser evidence.
