## Data Flow Trace

**Error Location:** `role-model-router/apps/runtime-host-bridge/src/validate-host.ts:118`

**Bad Value:** host never reaches a healthy listening state because its managed bridge subprocess never starts serving.

**Call Stack Trace:**
1. `waitForOk()` at `validate-host.ts:60-74` waits for host `/health`
2. `runRuntimeHostValidation()` at `validate-host.ts:86-119` launches llama-swap and assumes host startup
3. llama-swap `startRoleModelBridgeProcess()` at `rolemodel_bridge_process.go:133-148` launches the managed bridge and blocks on bridge `/healthz`
4. `buildRoleModelBridgeCommand()` at `rolemodel_bridge_process.go:65-90` invokes `corepack pnpm ... exec tsx src/cli.ts`
5. `src/cli.ts` defines `main()` but does not execute it; `src/cli-entry.ts` is the file that actually does `void main();`

**Root Cause:** the caller launches a library-style module (`src/cli.ts`) instead of the executable entrypoint (`src/cli-entry.ts`), so the bridge process exits successfully without starting a server.

### Hypothesis 1
**Statement:** `runtime:validate-host` times out because `rolemodel_bridge_process.go` launches `tsx src/cli.ts`, which exits immediately because `main()` is never invoked.

**Rationale:** direct execution of `tsx src/cli.ts` exits immediately with no `/healthz`, while `tsx src/cli-entry.ts` with identical arguments stays up and serves health.

**Confirmed Root Cause:** entrypoint mismatch in `buildRoleModelBridgeCommand()`

## Root Cause Summary

**Root Cause:** the llama-swap managed bridge launcher invokes `src/cli.ts` instead of the executable shim `src/cli-entry.ts`.

**Location:** `role-model-router/vendor/llama-swap/rolemodel_bridge_process.go:73`

**Detailed Explanation:** `src/cli.ts` now exports `main()` and helper functions but does not execute `main()` at module load. The executable behavior lives in `src/cli-entry.ts`, which imports `main` and calls `void main();`. Because llama-swap still shells into `tsx src/cli.ts`, the subprocess exits cleanly without binding a port, so llama-swap waits for `/healthz` until timeout, and the outer host validator then times out on host `/health`.

**Fix Strategy:** update the managed bridge launcher to invoke `src/cli-entry.ts`, then rerun the targeted Go test and `runtime:validate-host`.

**Test Plan:**
1. Add/adjust a failing unit test in `rolemodel_bridge_process_test.go` that expects the command to use `src/cli-entry.ts`.
2. Update `buildRoleModelBridgeCommand()` to use the executable entrypoint.
3. Rerun the targeted Go test.
