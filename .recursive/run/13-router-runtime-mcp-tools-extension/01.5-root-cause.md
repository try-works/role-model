## Pattern Analysis

**Working example:** existing validator commands that execute source entrypoints with `tsx`, where `.js` specifiers can resolve to sibling `.ts` files.

**Broken example:** `node dist/cli.js`, where plain Node uses package exports directly and expects actual `.js` files at the resolved source path.

| Aspect | Source-mode validator path | Compiled runtime path |
| --- | --- | --- |
| Entry file | `src/*.ts` | `dist/*.js` |

| Loader behavior | TS-aware remapping | plain Node ESM |
| Sibling import resolution | `.js` -> `.ts` works | `.js` must physically exist |

**Likely cause:** the package-export convention currently favors source-executed development commands but not compiled runtime execution.

## Root Cause Summary

**Root Cause:** runtime dependency package manifests do not declare built-runtime export paths, so plain Node resolves the compiled host executable back into source TypeScript modules.

**Location:** workspace `package.json` export maps in the runtime-host-bridge dependency graph, starting with `/role-model-router/packages/core/package.json`.

**Detailed Explanation:** the repo’s current source-first package-export pattern works for validators that run under `tsx`, because the loader can map `.js` specifiers back onto `.ts` sources. But the run-13 executable proof launches `dist/cli.js` under plain Node. Once that process crosses a workspace package boundary, Node follows `exports` and lands in `src/index.ts`, which immediately imports sibling `.js` files that are not present in `src/`. The failure is therefore a package-resolution mismatch, not a tool-execution, MCP connector, routing, or host-lifecycle defect.

**Fix Strategy:** add a dedicated `runtime` export condition pointing to `dist` for the runtime executable dependency graph, preserve current source-default behavior for `tsx`-based commands, and run the compiled CLI with `node --conditions=runtime`.

**Test Plan:** keep the executable export-contract test green, build the runtime graph, run the compiled CLI with `node --conditions=runtime`, and confirm the runtime host starts and serves the expected endpoints.
