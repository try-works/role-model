**Failure location:** `/role-model-router/apps/runtime-host-bridge/src/validate-host.ts:91`

**Bad value / bad assumption:** the validation path assumes `go run .` will start the vendored host successfully on a clean worktree and reach `/health`, but the vendored Go package assumes `proxy/ui_dist` already exists locally.

**Call path:**

## Root Cause Summary

**Root Cause:** the clean merged runtime baseline is not reproducible because the vendored `llama-swap` host requires generated `proxy/ui_dist` assets via `//go:embed ui_dist`, but those assets are ignored and absent in a fresh worktree.

**Location:** `/role-model-router/vendor/llama-swap/proxy/ui_embed.go`, `/role-model-router/vendor/llama-swap/proxy/.gitignore`, `/role-model-router/vendor/llama-swap/ui-svelte/vite.config.ts`

**Detailed Explanation:** run-11's host and observability validation receipts assumed a vendored host that could start with `go run .`, but the clean run-12 baseline proves that assumption was only true in a worktree where `proxy/ui_dist` already existed as leftover generated output. Because the validator waits on `/health` and does not emit captured host stderr on failure, the real startup error was hidden until the vendored host was run directly. The defect is therefore not a networking, bridge, or health-route bug; it is an undeclared generated-asset dependency in the vendored host startup path.

**Fix Strategy:** make the host-integrated validation path deterministic on a clean worktree by removing or satisfying the vendored `ui_dist` build-time dependency explicitly, and improve the validator so host startup stderr/stdout is surfaced when startup fails.

**Test Plan:** add or update red/green coverage so a fresh worktree can run the host-integrated validation path without relying on pre-existing ignored assets, and confirm that host validation failures emit actionable startup evidence instead of only timing out on `/health`.
